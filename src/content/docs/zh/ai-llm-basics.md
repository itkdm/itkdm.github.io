## 一、什么是 LLM（大语言模型）？

### 1.1 通俗理解

**LLM = Large Language Model（大语言模型）**

用一个比喻理解：
- **传统程序** = 计算器
  - 你输入 `2+2`，它输出 `4`
  - 规则是程序员写死的，不会变

- **LLM** = 读过很多书的助手
  - 它读过互联网上几乎所有的文本
  - 你问问题，它根据读过的内容组织答案
  - 不是"查数据库"，而是"理解后生成"

### 1.2 LLM 的核心能力

| 能力 | 描述 | 例子 |
|------|------|------|
| **文本生成** | 根据输入生成连贯的文本 | 写文章、写代码、写邮件 |
| **问答** | 回答问题 | "中国首都是哪里？" → "北京" |
| **翻译** | 跨语言转换 | 中文 → 英文 |
| **总结** | 提取核心信息 | 长文章 → 摘要 |
| **推理** | 逻辑推导 | "A>B, B>C, 所以 A>C" |

### 1.3 主流 LLM 对比

| 模型 | 公司 | 特点 | 适用场景 |
|------|------|------|---------|
| **GPT-4 / GPT-4o** | OpenAI | 综合能力最强，多模态 | 通用场景 |
| **Claude 3.5** | Anthropic | 长上下文，安全性好 | 文档分析、代码 |
| **Gemini 1.5** | Google | 多模态强，Google 生态 | 多模态任务 |
| **Qwen2.5** | 阿里 | 中文能力强，开源 | 中文场景 |
| **DeepSeek** | 深度求索 | 代码能力强，免费 | 编程辅助 |

## 二、什么是 Token？

### 2.1 Token 是什么？

**Token 是 LLM 处理文本的最小单位。**

可以理解为：
- 英文：≈ 4 个字母 或 3/4 个单词
- 中文：≈ 1-2 个汉字

```
英文示例：
"Hello world" → ["Hello", " world"]  (2 tokens)

中文示例：
"你好世界" → ["你", "好", "世", "界"]  (4 tokens)
或
"你好世界" → ["你好", "世界"]  (2 tokens，取决于分词)
```

### 2.2 Token 计数规则

| 语言 | 估算方式 | 例子 |
|------|---------|------|
| **英文** | 1000 tokens ≈ 750 单词 | "Hello world" ≈ 2 tokens |
| **中文** | 1000 tokens ≈ 600-800 汉字 | "你好世界" ≈ 2-4 tokens |
| **代码** | 1000 tokens ≈ 50-100 行 | 取决于语言和复杂度 |

**在线 Token 计数器：**
- https://platform.openai.com/tokenizer
- https://www.tiktokenizer.com

### 2.3 为什么 Token 重要？

| 原因 | 说明 |
|------|------|
| **计费单位** | API 按 Token 收费（输入 + 输出） |
| **速度影响** | Token 越多，生成越慢 |
| **上下文限制** | 模型有最大 Token 限制 |

**计费示例（GPT-4）：**
```
输入：1000 tokens → $0.03
输出：500 tokens → $0.06
总计：$0.09
```

## 三、什么是上下文窗口（Context Window）？

### 3.1 定义

**上下文窗口 = 模型一次能处理的最大 Token 数量**

包括：
- 你的输入（Prompt）
- 模型的输出（Completion）
- 对话历史（多轮对话时）

### 3.2 主流模型的上下文窗口

| 模型 | 上下文窗口 | 相当于 |
|------|-----------|--------|
| **GPT-4 Turbo** | 128K tokens | ≈ 300 页纸 |
| **Claude 3.5** | 200K tokens | ≈ 500 页纸 |
| **Gemini 1.5 Pro** | 1M tokens | ≈ 2500 页纸 |
| **Qwen2.5** | 32K-128K tokens | ≈ 75-300 页纸 |

### 3.3 上下文窗口的实际意义

```
场景：分析一份 100 页的 PDF 文档

❌ 上下文窗口太小（如 4K）：
   - 只能一次读几页
   - 需要分段处理，可能丢失整体信息

✅ 上下文窗口大（如 200K）：
   - 可以一次性读完整份文档
   - 模型能理解全文脉络
   - 回答更准确
```

### 3.4 超出上下文窗口怎么办？

| 方案 | 描述 | 优缺点 |
|------|------|--------|
| **截断** | 只保留最近的内容 | 简单，但丢失历史信息 |
| **摘要** | 把历史对话压缩成摘要 | 保留核心，但丢失细节 |
| **RAG** | 用检索增强生成 | 效果好，但实现复杂 |
| **滑动窗口** | 保留最近 N 轮对话 | 平衡方案和效果 |

## 四、Temperature（温度）参数

### 4.1 什么是 Temperature？

**Temperature 控制输出的随机性/创造性。**

```
Temperature = 0    →  最确定，每次输出几乎一样
Temperature = 0.7  →  平衡，推荐默认值
Temperature = 1.0  →  最有创意，但可能不稳定
```

### 4.2 不同场景的推荐值

| 场景 | 推荐 Temperature | 原因 |
|------|-----------------|------|
| **代码生成** | 0.0 - 0.2 | 需要准确，不要随机 |
| **数学计算** | 0.0 | 答案唯一，不能随机 |
| **客服问答** | 0.3 - 0.5 | 稳定但略有变化 |
| **创意写作** | 0.7 - 0.9 | 需要创意和多样性 |
| **头脑风暴** | 0.9 - 1.0 | 最大化创意 |

### 4.3 实际效果对比

```
Prompt: "写一个关于 AI 的短故事开头"

Temperature = 0.0:
"在 2050 年，人工智能已经普及到生活的每一个角落..."
（每次都是这个开头）

Temperature = 0.8:
"2050 年的某个清晨，AI 管家轻柔地唤醒了我..."
（每次可能不同，更有创意）
```

## 五、其他关键参数

### 5.1 Top-P（Nucleus Sampling）

**控制候选词的范围。**

- Top-P = 1.0：考虑所有可能的词
- Top-P = 0.9：只考虑累积概率前 90% 的词

**与 Temperature 的区别：**
- Temperature：调整概率分布的"陡峭度"
- Top-P：直接砍掉低概率的候选词

### 5.2 Max Tokens

**限制最大输出长度。**

```
设置：max_tokens = 500
效果：模型最多输出 500 tokens，然后停止

用途：
- 控制成本
- 防止生成太长
- 确保响应时间
```

### 5.3 Frequency Penalty / Presence Penalty

| 参数 | 作用 | 推荐值 |
|------|------|--------|
| **Frequency Penalty** | 降低重复词的频率 | 0-0.5 |
| **Presence Penalty** | 鼓励提到新话题 | 0-0.5 |

**用途：** 防止模型啰嗦、重复

## 六、LLM 工作原理（简化版）

### 6.1 训练过程

```
┌─────────────────────────────────────────────────────────────┐
│                    LLM 训练流程                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 收集数据                                                 │
│     互联网文本、书籍、代码、对话记录...                       │
│                                                             │
│  2. 预处理                                                   │
│     清洗、分词（Tokenize）、格式化...                        │
│                                                             │
│  3. 预训练                                                   │
│     预测下一个 Token，学习语言规律                           │
│     （这一步最花钱，几百万到上亿美元）                        │
│                                                             │
│  4. 微调（Fine-tuning）                                     │
│     用高质量数据优化特定能力                                 │
│                                                             │
│  5. 对齐（RLHF）                                            │
│     让人类评分，训练模型符合人类偏好                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 6.2 推理过程

```
用户输入："今天天气不错，我想去"

模型思考：
1. 把输入转成 Tokens: ["今天", "天气", "不错", "，", "我", "想", "去"]
2. 计算下一个 Token 的概率分布:
   - "公园" (35%)
   - "散步" (25%)
   - "旅行" (20%)
   - "游泳" (10%)
   - 其他 (10%)
3. 根据 Temperature 采样一个 Token
4. 把新 Token 加入输入，重复步骤 2-4
5. 直到生成结束符或达到 max_tokens

输出："今天天气不错，我想去公园散步。"
```

## 七、LLM 的局限性

### 7.1 幻觉问题

**模型会"一本正经地胡说八道"。**

```
问："2025 年诺贝尔物理学奖得主是谁？"

模型可能回答："2025 年诺贝尔物理学奖授予了张三教授，
以表彰他在量子计算领域的贡献。"

事实：2025 年的奖项还没颁发，模型在编造。
```

**解决方案：**
- 用 RAG 提供真实依据
- 让模型标注不确定性
- 人工核实关键信息

### 7.2 知识截止时间

| 模型 | 知识截止 |
|------|---------|
| GPT-4 Turbo | 2024 年 4 月 |
| Claude 3.5 | 2024 年 4 月 |
| Qwen2.5 | 2024 年 6 月 |

**解决方案：** 用 RAG 或搜索插件获取最新信息

### 7.3 计算能力有限

**LLM 不擅长：**
- 复杂数学计算
- 精确的逻辑推理
- 需要多步验证的任务

**解决方案：**
- 用工具（计算器、代码执行）
- 让模型分步思考（Chain of Thought）

## 🎯 面试回答版本

> 面试官问："请介绍一下 LLM 的基础概念" 或 "什么是 Token？"

### 标准回答框架（2 分钟）

```
LLM 是大语言模型，核心概念我主要从几个方面理解：

【Token】
Token 是模型处理文本的最小单位。英文约 4 个字母一个 token，
中文约 1-2 个汉字一个 token。Token 很重要，因为：
1. API 按 token 计费
2. 模型有最大 token 限制
3. 影响生成速度

【上下文窗口】
上下文窗口是模型一次能处理的最大 token 数。
比如 GPT-4 Turbo 是 128K，Claude 3.5 是 200K。
窗口越大，能处理的文档越长，但成本也越高。

【Temperature】
Temperature 控制输出的随机性。0 最确定，1 最有创意。
代码生成用 0-0.2，创意写作可以用 0.7-0.9。

【局限性】
LLM 有幻觉问题，可能编造信息；知识有截止时间；
不擅长精确计算。实际应用中常用 RAG 来解决这些问题。
```

### 高频追问

| 追问 | 参考回答 |
|------|---------|
| "Token 怎么计数？" | 英文约 750 词=1000 tokens，中文约 600-800 字=1000 tokens。可以用 OpenAI 的 tokenizer 工具精确计算。 |
| "如何减少 Token 消耗？" | 1) 精简 Prompt 2) 用摘要代替原文 3) 设置 max_tokens 限制 4) 用更小的模型 |
| "上下文窗口不够怎么办？" | 1) 截断历史 2) 摘要压缩 3) 用 RAG 检索关键信息 4) 换更大窗口的模型 |
| "幻觉问题怎么解决？" | 1) RAG 提供真实依据 2) 让模型标注引用 3) 关键信息人工核实 4) 用搜索插件验证 |
